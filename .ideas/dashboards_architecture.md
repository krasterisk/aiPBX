# Архитектура и концепция работы с дашбордами

В проекте aiPBX раздел дашбордов строится на принципах FSD (Feature-Sliced Design) и активно переиспользует общие UI компоненты и состояние фильтров. Ниже приведено описание основных методов, компонентов и путей прохождения данных.

## 1. Страницы (Pages)
Дашборды представлены тремя основными страницами, каждая из которых подсвечивает свой аспект аналитики, но использует общий слой фильтрации.

- **`DashboardOverviewPage`** (Сводный дашборд)
  - **Описание:** Главная страница статистики. Показывает агрегированную информацию о звонках (длительность, стоимость, количество).
  - **API:** Запросы идут к `useDashboard()` → `GET /api/reports/dashboard`
  - **Компоненты:** `DashboardStatistics`, `DashboardCharts`.  

- **`DashboardCallRecordsPage`** (Аналитика проектов / Operator Analytics)
  - **Описание:** Аналитика по конкретным проектам операторов с детальными метриками качества речи, эмоций, вежливости и AHT.
  - **API:** Запросы идут к `useGetOperatorDashboard()` → `GET /api/operator-analytics/dashboard`
  - **Компоненты:** `OperatorDashboard`, `DashboardBuilder`.

- **`AIAnalyticsPage`** (AI Аналитика)
  - **Описание:** Аналитика работы AI-ассистентов (например, интенты, качество работы нейросети, Token Usage).
  - **API:** Запросы идут к `useGetAIAnalyticsDashboard()` → `GET /api/ai-analytics/dashboard/data`
  - **Состояние:** Использует собственный слайс `aiAnalyticsPageSlice`.

## 2. Общий Слой (Widgets & Features)
Чтобы не дублировать код между разными страницами дашбордов, используется общая обертка и общее состояние.

### Widget: `DashboardLayout`
- **Роль:** Общая обертка-лейаут для любой страницы дашборда. 
- Отрисовывает заголовок страницы и блок фильтров (`FiltersGroup`). Содержимое самой страницы передается через `children`.
- Данные для фильтров читаются напрямую из Redux-слайса фичи `Dashboard`.

### Feature: `Dashboard`
Здесь хранится вся бизнес-логика состояния фильтров и общие чарты.

- **Стейт-менеджмент (`dashboardPageSlice.ts`)**
  - Подключается асинхронно через `DynamicModuleLoader`.
  - Хранит глобальное состояние фильтров для дашбордов: `startDate`, `endDate`, `userId`, `assistantId`, `source`, `tab`.
  - Эти данные реактивно управляют всеми графиками, независимо от страницы.

- **Хук `useDashboardFilters.ts`**
  - Общий хук, который *не только* читает стейт, но и инициирует RTK-запрос `useDashboard()` к `/reports/dashboard`. 
  - **Важно:** Этот хук нужно вызывать только на тех страницах, где реально требуются данные агрегированного сводного отчета. Для других страниц (как "Аналитика проектов") фильтры следует читать напрямую селекторами (например, `getDashboardUserId`, `getDashboardStartDate` и т.д.), чтобы не запускать лишние запросы.

- **UI Компоненты Фичи:**
  - `FiltersGroup` — Шапка с выбором периода (Сегодня, Неделя, Месяц и т.д.) и кнопкой открытия расширенного фильтра.
  - `PeriodExtendedFilters` — Расширенные фильтры (выбор клиента, ассистента, конкретных дат и источника). При выборе клиента `ClientSelect` кидает событие, которое обновляет `userId` в Redux, что в свою очередь триггерит обновление дашбордов.

## 3. Этапы прохождения данных (Data Flow)
Вот как меняется состояние и обновляются данные на примере выбора клиента:

1. **Взаимодействие с UI:** Пользователь в `PeriodExtendedFilters` выбирает клиента через `ClientSelect`.
2. **Экшен:** `ClientSelect` вызывает `onChangeClient`, который внутри `DashboardLayout` диспатчит экшен `dashboardPageActions.setUserId('123')`.
3. **Обновление Redux:** Слайс `dashboardPage` обновляет поле `userId`.
4. **Реактивность (Фронтенд):** 
   - `useSelector(getDashboardUserId)` на активной странице дашборда детектит изменение.
   - Хук страницы обновляет аргументы в вызове RTK Query (например, `useGetOperatorDashboard({ userId: '123' })`).
5. **Запрос к серверу (RTK Query):** RTK Query видит, что аргументы `userId` поменялись, и автоматически шлет новый `GET` запрос к соответствующему эндпоинту с нужным query-параметром.
6. **Рендеринг:** После получения новых данных, компоненты (например, `OperatorDashboardId`) перерисовывают графики (`MuiCharts`).

## 4. Паттерны, на которые стоит опираться
- **Deduplication в RTK Query:** Если 2 компонента одновременно вызывают хук (например, один и тот же `useDashboard`), RTK Query смёрджит их в один сетевой запрос. Но подписок будет 2. 
- **Разделение Фильтров и Данных:** Лейаут дашборда требует только фильтры. Поэтому `DashboardLayoutInner` читает стейт напрямую из селекторов (без вызова `useDashboardFilters`). Страница же вызывает API-запрос (через RTK Query) используя те же селекторы. 
- **Поддержка Ролей:** Глобальное вычисление `userId` для запросов учитывает роль пользователя (`isAdmin`). Если юзер — клиент, его `userId` подставляется из `authData`. Если админ — используется `clientId` из стейта, выбранный через `ClientSelect`.
